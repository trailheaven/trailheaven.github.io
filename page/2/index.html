ガントチャートの決め手は平準化

2016-12-11
WBS, ライトイールド

　ガントチャートツールを選ぶ際、私は「平準化」が行えるかを基準に考えます。ということで、今回は平準化とは何なのか、また何がそんなに嬉しいのかについて書いてみたい思います。

ガントチャートの平準化

  1. 平準化とは
  2. ガントチャートの平準化が行うこと
  3. 平準化はいつ、何のために使うのか
  4. 平準化を使うメリットとは
  5. 平準化が使えるガントチャートツール

1. 平準化とは

まずは平準化の定義から。

　需要や業務負荷などが特定の時期や資源に集中しているといった偏りを排し、均等に配分すること。

　製造業で生産品種ごとにロット生産を行った場合、生産中は生産設備や人員はフル操業となるが、その品種を生産していない期間も設備や在庫を持つことになる。トヨタ生産方式ではこれを“ムラ”と考え、生産の平準化を推奨・実施している。

　プロジェクトマネジメントなどでは、プロジェクトの計画段階で特定の人員にタスクが集中してしまうことがあるが、このままでは実施不可能である。そこでリソースヒストグラムを利用して、作業の平準化を行う。

　業務システムにおいては、アプリケーションごとにサーバを立てると「昼間はアクセスが集中するが、夜間は遊休化してしまう」ことが多い。そこで仮想化技術によって、業務アプリケーションサーバと夜間バッチサーバでサーバ資源を共有化し、サーバ負荷の平準化を行うといったソリューションが推奨されている。

(ITmedia 情報マネジメント用語辞典より)

　大きな括りの言葉ではありますが、これから説明する平準化とは、「プロジェクトメンバーの作業を平すこと」を指します。

2. ガントチャートの平準化が行うこと

　ガントチャートツールの平準化とは一体どんな機能なのか。まずは平準化を行う前後のガントチャートを見て下さい。

　平準化を行うことで、メンバーの作業が平されたのが分かります。これは予め設定された1日の作業時間を超えたためであり、結果、コンピューターはメンバーの負荷を減らしつつも、プロジェクトの期間が長くなるように変更しています。つまり、平準化とはメンバーの負荷を考えながら、プロジェクト期間をより現実的なものに変えてくれる機能なのです。

3. 平準化はいつ、何のために使うのか

　平準化を使うタイミングは「いつでも」です。ガントチャートの作業を追加した、変更した、削除した等々、ガントチャートに対して変更を加えた時は常に平準化を行います。では何のために使うのか、それは「未来を予測するため」です。

　私はガントチャートツールを「コンピューターに未来を予測してもらうツール」と捉えています。コンピューターに未来を予測してもらうことで、これからのスケジュールが客観的に見られるからです。自分で考えると、自分の守備範囲でしか計算が行われません。特に期待値などを含む個人的感情で物事の優先順位が決まりがちで、結果として実態とはズレた精度の低いスケジュールになってしまいます。

　では、平準化のないガントチャートツールで計画を立てるとどのような未来になるのでしょうか？答えは簡単です、自分が頭に描いた未来予測にしかなりません。自分の考え方が正しいことを前提にした、自分にとって気持ちの良い（キレイな）ガントチャートが生まれるだけです。言い換えるなら、自分にとって都合良く作られたスケジュールが生まれるだけです。

4. 平準化を使うメリット

　先述の通り、平準化を使うメリットはコンピューターが計算することにあります。自分で考えるのではなく、コンピューターに考えてもらうことで、何度も繰り返し計算を行うことができます。時にはコンピューターが計算することで自分の予想と違う結果も生まれますが、これも設定されたパラメータが間違っていなければ生まれるべくして生まれた計画です。

　平準化を嫌う人の多くは、この予想と違う結果が生まれることを嫌います。つまり、「自分の思う通りにならないから」という理由ですが、それはどうでしょうか。私は自分が正しいと過信するのではなく、コンピューターの考えた結果を選択肢の一つとして検討すべきと考えます。

　実際、私がスケジュールを計画する際は、パラメータの設定に一番時間を使います。そして、後はコンピュータが計算した結果を採用するようにしています。ただし、結果を採用するまでの間、パラメータを細かく変更しつつ、どのような状態でプロジェクトを運営するのが一番効率の良い方法か、をとことんシミュレーションします。

5. 平準化が使えるガントチャートツール

　世の中にはガントチャートツールがたくさんありますが、平準化の行える機能は意外とありません。私が知る限り以下の２択しか無く、それもOSによってどちらかが決定します。

  • Microsoft Project
    　言わずと知れたガントチャートの王道です。私自身このツールが大好きで、前職では10年近くお世話になりました。多彩なパラメータをもとにする平準化は複雑で、未来予測のシミュレーションを行うには打ってつけです。クラウド版、デスクトップ版とアプリケーションの形態も多岐に渡るのですが、残念ながら平準化はデスクトップ版にしか実装されていないようです。Macユーザーにとっては敷居が高いため、現在は利用していません。

  • Omni Plan
    　Macユーザー向けのガントチャートツールです。StandardはMacOS版とiOS版の２種類展開で、ProはMacOS版のみの展開です。平準化はStandardに含まれていますので、私は値段の安いこちらを使用しています。このアプリの特徴は、UIでしょうか。デザインが綺麗ですし、操作も直感的に行えるように工夫されています。一方、残念な点もあります。不具合に近い症状なのですが「保存していないのに上書き保存される」「元の状態に戻ろうと操作を取り消しても、変な状態に戻ってしまう」などなど。品質にいささかの疑問がありますので、使用する際には必ず作業前のバックアップを取るなどの注意が必要です。なおアプリには日本語版仕様がありますが、サポートは英語です。日本語で問い合わせると、翻訳ツールを通したと思われる難解な日本語で返事が返ってきます。

終わりに

　いかがでしたか。ガントチャートツールはスケジュールを管理するのではなく、未来を予測するツール。そう考えると、これまでとは少し違った目線で見ることができるのではないかと思います。私はWBSに変更があればすぐさまガントチャートツールで平準化を行い、こらからのスケジュールを変更します。当初計画からの変更を嫌う人がいますが、私は「過去の計画に捉われず、現在の状況を正確に把握し、未来を客観的に予測する」を繰り返すことがプロジェクト管理のキモだと考えています。

　平準化を使ったことがない人、平準化が苦手な人にとって参考になれば幸いです。ではまた▷L◁

WBSの勘所（５選）

2016-11-23
WBS, ライトイールド

WBSの勘所として、私がWBSを作る際に意識している点をまとめてみました。

WBSの勘所

  1. プロジェクトのゴールを定義する
  2. ゴールまでの作業を洗い出す
  3. 作業を単位当たり１人日程度にバラす
  4. 作業の依存関係を設定する
  5. WBSを随時更新する

WBSとは

まずはWBSの定義から。

WBSとは、プロジェクトマネジメントで計画を立てる際に用いられる手法の一つで、プロジェクト全体を細かい作業に分割した構成図。

WBSでは、まずプロジェクトの成果物をできるだけ細かい単位に分解していく。その際、全体を大きな単位に分割してから、それぞれの部分についてより細かい単位に分割していき、階層的に構造化していく。成果物の細分化が終わったら、それぞれの部分を構成するのに必要な作業（一つとは限らない）を考え、最下層に配置していく。個々の部分を構成する一連の作業のかたまりのことを「ワークパッケージ」と呼ぶ。WBSのそれぞれのワークパッケージに担当する人員を配置していけば、プロジェクトを遂行する組織図ができる。これをOBS（Organization Breakdown Structure）と呼ぶ。(IT用語辞典 e-Wordsより)

1. プロジェクトのゴールを定義する

　「プロジェクト全体を細かい作業に分割したものをWBSと呼ぶ」と定義されている通り、WBSはプロジェクトがあってこその構成図です。そのため、まずはプロジェクトのゴールを設定するところから始めます。

　と言っても難しいことはせず、ただ単純に関係者に対してプロジェクトの目的やゴールをどう考えているかを聞き、理想的なゴールを定義するイメージです。

　例えば「カレーライスを作る」と言うプロジェクトがあった場合、そのゴールは「カレーライスを作る」までなのか、「カレーライスをテーブルに並べる」までなのか、「後片付けをする」までなのかを決めると言った感じです。

2. ゴールまでの作業を洗い出す

　プロジェクトのゴールが決まれば、ゴールまでに必要な作業を洗い出します。ここでは作業の大小や前後関係を考えず、ブレーンストーミングさながら思いつくままにリストアップします。どんなに大雑把でも、無理難題な作業でもいいので、完了までの道を書きます。詳細が不明確な作業も現時点で要否を判断せず、リストに含めます。

　大事なことは 「ゴールまでの道を想像しながら作業を書き出す」ことです。当たり前ですが、ゴールまでの道が書けなければプロジェクトがゴールすることはありません。粒度は荒くてもいいので、このタイミングで必ずシミュレーションを行い、ゴールまでの道を書くようにします。

　もし「カレーライスを作る」プロジェクトで「ご飯を作る」作業が抜けてしまった場合、「カレーは飲み物か」問題が発生して大変です。

3. 作業を単位当たり１人日程度にバラす

　ゴールまでの作業を洗い出した後はこれらリストを細かく分解します。目標は１作業当たり１人日程度の作業ボリュームです。１人日程度のボリュームであれば、その作業が妥当かどうかを判断しやすいからです。作業工数が大きければ大きくなるほど、その数字が妥当なのかどうか判断が難しくなります。

　もちろん「そうは言っても…」という作業は往往にしてあります。その場合は潔く諦め、無理せずそにのままにしておきます。１人日はあくまでも目標値です。

　「カレーライスを作る」プロジェクトは恐らく１人日以下のプロジェクトでしょう。このような場合は作業の単位を「日」から「時間」に変更します。

4. 作業の依存関係を設定する

　作業が出揃った後は、各作業の依存関係を設定します。作業によっては他の作業が完了しないと着手できない場合もありますし、他の作業と同時に着手できる作業もあります。これら依存関係はプロジェクトの作業期間に影響するため、可能な限り洗い出すようにします。

　「カレーライスを作る」プロジェクトの「ご飯を作る」と「カレールーを作る」は同時に開始することができますが、「お皿に入れる」作業はこれら２つの作業が完了していないと着手できません。

5. WBSを随時更新する

　WBSは「生き物」です。日々刻々と変化する状況の中、時間の経過とともに作業が変わるのは世の常です。今まであやふやだった作業も着手前になれば明確になります。私の場合、作業に変化が生じると即座にWBSへ反映させるようにしています。そのため、私が作成するWBSはプロジェクト開始前に作ったものは作業の粒度が荒く、時間の経過と共に粒度が細かくなり、プロジェクトが完了する頃には全てが実際に行った作業に置き変わる感じです。

　ただし、人月ベースの作業見積もりを行う場合は、プロジェクト開始前に作ったWBSが使われることが一般的です。それが原因で、プロジェクト開始前のWBSは往々にして一人歩きします。更にはWBSの変更が簡単に受け入れられないケースもありますので注意が必要です。WBSを随時変更すると「計画的に実行できていない」と見られるんですね。私には「プロジェクトが柔軟に対応できて健康的」と見えるのですが。

　とは言え、「カレーライスを作る」プロジェクトでWBSが頻繁に更新される場合、単位が時間ですので流石に「計画的に実行できていない」と言われても仕方がありませんね…。

終わりに

いかがでしたか。以上に挙げた勘所が皆さんの参考になれば幸いです。
